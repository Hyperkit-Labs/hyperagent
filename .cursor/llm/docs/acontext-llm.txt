# Acontext for HyperAgent

Acontext is a **context data platform for self-learning agents**: a unified backend for storing, observing, and learning from agent context (messages, tool calls, artifacts). [github](https://github.com/memodb-io/Acontext)
HyperAgent uses Acontext as its **long-term memory layer** so agents can remember prior runs, analyze failures, and distill successful strategies.

***

## Roles in HyperAgent

- **Agent memory**  
  - Persist user requirements, design decisions, and resolved issues across sessions.

- **Context management**  
  - Provide structured conversation and task history for each workspace/project, not just raw chat logs.

- **Experience learning**  
  - Store successful tool-call chains and outcomes as “experiences” that can be retrieved and reused. [acontext](https://acontext.io/blog/introducing-acontext-context-data-platform-for-self-learning-ai-agents)

- **Observability**  
  - Log traces of runs (inputs, outputs, tools, status) for debugging agent behavior. [jimmysong](https://jimmysong.io/ai/acontext/)

- **Knowledge retention**  
  - Maintain a durable knowledge base for each agent type (SpecAgent, CodeGenAgent, AuditAgent), keyed by workspace and project.

## Core concepts (mapped to HyperAgent)

Acontext uses structured models (Sessions / Spaces / Artifacts, or similar) to organize context. [jimmysong](https://jimmysong.io/ai/acontext/)

For HyperAgent:

- **Workspace**  
  - Map to an Acontext “space” or top-level context.  
  - Key: `workspace_id`.

- **Session / Run**  
  - A single agent run or workflow execution (SpecAgent, CodeGenAgent, etc.).  
  - Key: `run_id`, `agent_type`, `timestamp`.

- **Message / Memory**  
  - Individual interaction or distilled memory (e.g., “User prefers gas-optimized contracts”).  
  - Includes content + metadata (workspace, agent, tags).

- **Artifacts**  
  - Linked objects like contract source, diffs, audit reports, stored in S3/IPFS/EigenDA, referenced from Acontext. [jimmysong](https://jimmysong.io/ai/acontext/)

## HyperAgent AcontextToolkit (design)

### Interface

```ts
// acontext/types.ts
export interface MemoryMetadata {
  workspaceId: string;
  agent: string;             // "SpecAgent" | "CodeGenAgent" | "AuditAgent" | ...
  runId?: string;
  tags?: string[];
  importance?: "low" | "medium" | "high";
}

export interface CreateMemoryInput {
  content: string;           // free-form text or JSON string
  metadata: MemoryMetadata;
}

export interface CreateMemoryOutput {
  memoryId: string;
}

export interface SearchMemoriesInput {
  workspaceId: string;
  agent?: string;
  query: string;
  limit?: number;
}

export interface MemoryRecord {
  id: string;
  content: string;
  metadata: MemoryMetadata;
  createdAt: string;
}

export interface SearchMemoriesOutput {
  memories: MemoryRecord[];
}

export interface AcontextToolkit {
  createMemory(input: CreateMemoryInput): Promise<CreateMemoryOutput>;
  searchMemories(input: SearchMemoriesInput): Promise<SearchMemoriesOutput>;
}
```

### Implementation (Node/TypeScript, REST API)

Assuming Acontext exposes an HTTP API similar to the docs. [acontext](https://acontext.io)

```ts
// acontext/client.ts
import fetch from "node-fetch";
import type {
  AcontextToolkit,
  CreateMemoryInput,
  CreateMemoryOutput,
  SearchMemoriesInput,
  SearchMemoriesOutput,
} from "./types";

const ACONTEXT_BASE_URL = process.env.ACONTEXT_BASE_URL!;
const ACONTEXT_API_KEY = process.env.ACONTEXT_API_KEY!;

function headers() {
  return {
    "Content-Type": "application/json",
    Authorization: `Bearer ${ACONTEXT_API_KEY}`,
  };
}

export function createAcontextToolkit(): AcontextToolkit {
  return {
    async createMemory(input: CreateMemoryInput): Promise<CreateMemoryOutput> {
      const res = await fetch(`${ACONTEXT_BASE_URL}/memories`, {
        method: "POST",
        headers: headers(),
        body: JSON.stringify({
          content: input.content,
          metadata: {
            workspace_id: input.metadata.workspaceId,
            agent: input.metadata.agent,
            run_id: input.metadata.runId,
            tags: input.metadata.tags ?? [],
            importance: input.metadata.importance ?? "medium",
          },
        }),
      });

      if (!res.ok) {
        const text = await res.text();
        throw new Error(`Acontext createMemory failed: ${res.status} ${text}`);
      }

      const json = (await res.json()) as { id: string };
      return { memoryId: json.id };
    },

    async searchMemories(input: SearchMemoriesInput): Promise<SearchMemoriesOutput> {
      const res = await fetch(`${ACONTEXT_BASE_URL}/memories/search`, {
        method: "POST",
        headers: headers(),
        body: JSON.stringify({
          query: input.query,
          workspace_id: input.workspaceId,
          agent: input.agent,
          limit: input.limit ?? 5,
        }),
      });

      if (!res.ok) {
        const text = await res.text();
        throw new Error(`Acontext searchMemories failed: ${res.status} ${text}`);
      }

      const json = (await res.json()) as { results: any[] };
      const memories: MemoryRecord[] = json.results.map((m) => ({
        id: m.id,
        content: m.content,
        metadata: {
          workspaceId: m.metadata.workspace_id,
          agent: m.metadata.agent,
          runId: m.metadata.run_id,
          tags: m.metadata.tags,
          importance: m.metadata.importance,
        },
        createdAt: m.created_at,
      }));

      return { memories };
    },
  };
}
```

from acontext import AcontextClient

client = AcontextClient(api_key=os.environ["ACONTEXT_API_KEY"])

def store_memory(content: str, workspace_id: str, agent: str, run_id: str | None = None):
  memory_id = client.create_memory(
      content=content,
      metadata={
          "workspace_id": workspace_id,
          "agent": agent,
          "run_id": run_id,
      },
  )
  return memory_id

def search_context(query: str, workspace_id: str, agent: str | None = None, limit: int = 5):
  results = client.search_memories(
      query=query,
      workspace_id=workspace_id,
      agent=agent,
      limit=limit,
  )
  return results

## How HyperAgent should use Acontext

### When to write

- After **successful workflows**:
  - SpecAgent stores a distilled summary: “User X prefers ERC-20 with EIP‑2612 permits.”  
  - CodeGenAgent stores patterns that compiled and passed audits.  
  - AuditAgent stores known vulnerabilities and their fixes, plus contextual tags.

- After **notable failures**:
  - Store trace + root cause summary, to avoid repeating mistakes.

The LLM should **not** dump entire raw transcripts as memories; instead, orchestrators extract concise, semantically useful entries.

### When to read

Before each agent run:

- Construct a `query` from current task (e.g., “gas optimization preferences for workspace ws-123”).  
- Call `searchMemories` with `workspaceId` and agent type.  
- Inject top‑k results into the system prompt or intermediate context as “prior knowledge”.

***

## Best practices

1. **Good metadata**  
   - Always include `workspace_id`, `agent`, `run_id`, and at least one tag (e.g., `["gas_optimization", "erc20"]`) to improve retrieval quality. [jimmysong](https://jimmysong.io/ai/acontext/)

2. **Separation of semantic vs episodic**  
   - Store both:
     - semantic facts (“User prefers Base mainnet”), and  
     - episodic references (“2025‑02‑01: Deployed X vault on Base”). [mem0](https://mem0.ai/blog/long-term-memory-ai-agents)

3. **Privacy / isolation**  
   - Enforce workspace scoping in both `createMemory` and `searchMemories`; never leak memories across tenants.

4. **Retention & pruning**  
   - Periodically consolidate many low‑importance memories into higher‑level summaries (experience distillation) to keep the store efficient. [mem0](https://mem0.ai/blog/long-term-memory-ai-agents)

5. **Observability**  
   - Log each memory write/read with `run_id` and link to your own tracing/metrics, so you can see how memory influences agent outcomes. [acontext](https://acontext.io/blog/introducing-acontext-context-data-platform-for-self-learning-ai-agents)
