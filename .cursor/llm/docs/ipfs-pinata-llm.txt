# IPFS & Pinata for HyperAgent

IPFS is a content-addressed, peer‑to‑peer data layer. Pinata is a managed IPFS pinning and gateway service. [docs.ipfs](https://docs.ipfs.tech)
HyperAgent uses IPFS/Pinata as a **verifiable storage backend** for artifacts, reports, and long‑term agent memory.

***

## Core roles in HyperAgent

- **Artifact storage**  
  - Store generated smart contracts, compiled artifacts, audit reports, and deployment metadata as immutable CIDs.  
- **Verifiable memory**  
  - Persist execution traces, analysis summaries, and RAG documents so future agent runs can reference them by CID.  
- **Decentralized persistence**  
  - Avoid single-node storage; allow external verification of outputs via IPFS gateways. [docs.ipfs](https://docs.ipfs.tech)
- **MCP integration**  
  - Use the Pinata MCP server to let agents upload/download files via a standard tool interface. [linkedin](https://www.linkedin.com/pulse/building-mcp-server-pinatas-api-pinatacloud-vs0rc)

***

## Key concepts

- **CID (Content Identifier)**  
  - Hash of the content; if bytes change, CID changes. [docs.ipfs](https://docs.ipfs.tech/quickstart/pin/)

- **Pinning**  
  - A promise by some node or service (Pinata) to keep a CID’s content available. [docs.ipfs](https://docs.ipfs.tech/quickstart/pin/)

- **Gateway**  
  - HTTP interface to fetch `https://<gateway>/ipfs/<cid>` with regular HTTP clients. [docs.ipfs](https://docs.ipfs.tech/reference/)

- **Pinset / groups**  
  - Collections of related CIDs (e.g., all artifacts for one workflow execution). [playbooks](https://playbooks.com/mcp/pinatacloud/pinata-mcp)

## HyperAgent storage architecture

- **services/storage**  
  - Responsible for writing files to local disk / object storage and pinning CIDs via Pinata.  
  - Tracks mapping of `workspace_id / run_id / artifact_type → CID`.

- **RAG / vector DB (Qdrant)**  
  - Stores embeddings + metadata, with `cid` as a stable reference to full content in IPFS.  

- **MCP tools (Pinata MCP)**  
  - Give agents controlled access to upload/download files:
    - Allowed directories only.  
    - `PINATA_JWT` and `GATEWAY_URL` configured via env. [linkedin](https://www.linkedin.com/pulse/building-mcp-server-pinatas-api-pinatacloud-vs0rc)

## IPFS & Pinata Toolkit (Node/TypeScript)

HyperAgent should talk to IPFS/Pinata through a small toolkit rather than raw fetch everywhere.

### Interface

```ts
// storage/ipfs/types.ts
export interface PinFileInput {
  path: string;                // local file path
  name?: string;               // human name / label
  metadata?: Record<string, any>;
}

export interface PinFileOutput {
  cid: string;
  size: number;
  timestamp: string;
}

export interface FetchByCidInput {
  cid: string;
  gatewayUrl?: string;         // override default
}

export interface FetchByCidOutput {
  content: string | Buffer;
}

export interface IpfsPinataToolkit {
  pinFile(input: PinFileInput): Promise<PinFileOutput>;
  fetchContent(input: FetchByCidInput): Promise<FetchByCidOutput>;
}
```

### Implementation with Pinata REST API

```ts
// storage/ipfs/pinataToolkit.ts
import fs from "fs";
import FormData from "form-data";
import fetch from "node-fetch";
import type {
  IpfsPinataToolkit,
  PinFileInput,
  PinFileOutput,
  FetchByCidInput,
  FetchByCidOutput,
} from "./types";

const PINATA_ENDPOINT = "https://api.pinata.cloud/pinning/pinFileToIPFS";
const DEFAULT_GATEWAY = process.env.PINATA_GATEWAY_URL ?? "https://gateway.pinata.cloud";

export function createIpfsPinataToolkit(): IpfsPinataToolkit {
  const jwt = process.env.PINATA_JWT;
  if (!jwt) throw new Error("PINATA_JWT env var is required");

  async function pinFile(input: PinFileInput): Promise<PinFileOutput> {
    const data = new FormData();
    data.append("file", fs.createReadStream(input.path));

    if (input.name || input.metadata) {
      data.append(
        "pinataMetadata",
        JSON.stringify({
          name: input.name ?? undefined,
          keyvalues: input.metadata ?? undefined,
        }),
      );
    }

    const res = await fetch(PINATA_ENDPOINT, {
      method: "POST",
      headers: {
        Authorization: `Bearer ${jwt}`,
      },
      body: data as any,
    });

    if (!res.ok) {
      const text = await res.text();
      throw new Error(`Pinata pinFile failed: ${res.status} ${text}`);
    }

    const json = (await res.json()) as {
      IpfsHash: string;
      PinSize: number;
      Timestamp: string;
    };

    return {
      cid: json.IpfsHash,
      size: json.PinSize,
      timestamp: json.Timestamp,
    };
  }

  async function fetchContent(input: FetchByCidInput): Promise<FetchByCidOutput> {
    const gateway = input.gatewayUrl ?? DEFAULT_GATEWAY;
    const url = `${gateway}/ipfs/${input.cid}`;

    const res = await fetch(url);
    if (!res.ok) {
      const text = await res.text();
      throw new Error(`Gateway fetch failed: ${res.status} ${text}`);
    }

    const contentType = res.headers.get("content-type") || "";
    if (contentType.includes("application/json") || contentType.startsWith("text/")) {
      return { content: await res.text() };
    } else {
      return { content: Buffer.from(await res.arrayBuffer()) };
    }
  }

  return { pinFile, fetchContent };
}
```

This matches Pinata’s official `pinFileToIPFS` endpoint and gateway usage. [github](https://github.com/dabit3/ipfs-pinata-example)


## Python examples (for workers)

### Pin file

```python
import requests

PINATA_JWT = os.environ["PINATA_JWT"]
PINATA_ENDPOINT = "https://api.pinata.cloud/pinning/pinFileToIPFS"

def pin_file(path: str, name: str | None = None, metadata: dict | None = None) -> dict:
    with open(path, "rb") as f:
        files = {"file": (path, f)}
        data = {}
        if name or metadata:
            data["pinataMetadata"] = json.dumps({
                "name": name,
                "keyvalues": metadata or {}
            })
        res = requests.post(
            PINATA_ENDPOINT,
            headers={"Authorization": f"Bearer {PINATA_JWT}"},
            files=files,
            data=data,
        )
    res.raise_for_status()
    return res.json()   # includes IpfsHash, PinSize, Timestamp
```

### Fetch via gateway

```python
import requests

GATEWAY = os.environ.get("PINATA_GATEWAY_URL", "https://gateway.pinata.cloud")

def fetch_ipfs_text(cid: str) -> str:
    url = f"{GATEWAY}/ipfs/{cid}"
    res = requests.get(url)
    res.raise_for_status()
    return res.text
```

## MCP / Agent Integration

When using the **Pinata MCP server** with agent clients (e.g., Claude Desktop), the config looks like: [playbooks](https://playbooks.com/mcp/pinatacloud/pinata-mcp)

```jsonc
{
  "mcpServers": {
    "pinata": {
      "command": "npx",
      "args": ["pinata-mcp", "/path/to/allowed/directories"],
      "env": {
        "PINATA_JWT": "<YOUR_JWT>",
        "GATEWAY_URL": "example.mypinata.cloud"
      }
    }
  }
}
```

Capabilities exposed to agents (conceptually):

- List local files under allowed directories.  
- Upload files to Pinata (pin to IPFS).  
- Search and fetch file metadata.  
- Create private HTTP gateway links.  
- Fetch content via IPFS gateway and save locally. [linkedin](https://www.linkedin.com/pulse/building-mcp-server-pinatas-api-pinatacloud-vs0rc)

HyperAgent should:

- Treat Pinata MCP as a **tool server** for file I/O.  
- Restrict directories and CIDs it can access.  
- Log every MCP operation per `run_id` for auditability.

## Best practices for HyperAgent

1. **Always pin important artifacts**  
   - Contract source, build artifacts, audits, and execution reports should all be pinned; store the CID in your DB row for that run. [ipfscluster](https://ipfscluster.io/documentation/reference/api/)

2. **Attach metadata**  
   - Use `pinataMetadata.keyvalues` for fields like `workspace_id`, `run_id`, `artifact_type`, `chain_id`, `contract_address`. [docs.pinata](https://docs.pinata.cloud/api-reference/endpoint/ipfs/pin-file-to-ipfs)

3. **Use dedicated gateway**  
   - Prefer a custom Pinata gateway subdomain (e.g., `https://hyperagent.mypinata.cloud/ipfs/<cid>`) for UX and rate-limiting control. [docs.ipfs](https://docs.ipfs.tech/reference/http/gateway/)

4. **Isolate agent access**  
   - When exposing IPFS/Pinata to LLM agents, enforce:
     - Allowed directories for uploads.  
     - Maximum file size.  
     - Limited sets of operations (no delete without explicit policy).

5. **Redundancy**  
   - For critical artifacts, consider pinning CIDs to multiple services or running your own IPFS node in addition to Pinata. [ipfscluster](https://ipfscluster.io/documentation/reference/api/)
