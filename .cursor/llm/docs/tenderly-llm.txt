# Tenderly

Tenderly is a full-stack Web3 development platform for **simulating, debugging, monitoring, and operating smart contracts** across EVM-compatible networks. [docs.tenderly](https://docs.tenderly.co)
It lets you preview transaction outcomes, inspect state changes, analyze gas usage, and run contracts against forked mainnet-style environments without sending real transactions. [github](https://github.com/Tenderly/tenderly-docs)

***

## Core concepts

- **Simulations**  
  Run single or bundled transaction simulations against live or forked network state to preview effects on balances, storage, logs, and gas without on-chain execution. [docs.tenderly](https://docs.tenderly.co/reference)

- **Forks / Virtual TestNets**  
  Create private mainnet replicas (Forks / Virtual TestNets) with custom state, infinite faucet, and synchronized data; use them as dev/staging environments for dapps and agents. [github](https://github.com/Tenderly/tenderly-docs)

- **Debugger & Gas Profiler**  
  Step through failed or successful transactions line-by-line, inspect call traces, decoded inputs/outputs, and analyze gas usage at a function level. [docs.moonbeam](https://docs.moonbeam.network/tutorials/eth-api/using-tenderly/)

- **Monitoring & Alerts**  
  Configure monitors and alerts on contract events, transactions, or state changes to detect anomalies and production issues in real time. [quicknode](https://www.quicknode.com/builders-guide/tools/transaction-simulation-by-tenderly?category=rpc-node-providers)

- **Node / RPC**  
  Use Tenderly Node RPC and custom endpoints to execute standard Ethereum JSON-RPC methods and integrate simulation into wallets or dapps. [quicknode](https://www.quicknode.com/builders-guide/tools/transaction-simulation-by-tenderly)

## SDK usage (TypeScript)

The Tenderly SDK is the preferred way to integrate simulations and forks into Node/TypeScript environments such as agent runtimes. [docs.tenderly](https://docs.tenderly.co/tenderly-sdk/intro-to-tenderly-sdk)

### Installation

```bash
npm install @tenderly/sdk
# or
yarn add @tenderly/sdk
# or
pnpm add @tenderly/sdk
```

You also need a **Tenderly Access Key**, account name, and project slug from the dashboard. [github](https://github.com/Tenderly/tenderly-docs/blob/master/simulations-and-forks/reference/configuration-of-api-access.md)

### Basic initialization example

```ts
import { Tenderly } from "@tenderly/sdk";

const tenderly = new Tenderly({
  accountName: process.env.TENDERLY_ACCOUNT!,   // or organization name
  projectSlug: process.env.TENDERLY_PROJECT!,   // project slug from dashboard
  accessKey: process.env.TENDERLY_ACCESS_KEY!,  // API access key
  network: "mainnet",                           // or specific chain id/alias
});
```

Required environment variables typically include:

```env
TENDERLY_ACCOUNT=your-account-or-org
TENDERLY_PROJECT=your-project-slug
TENDERLY_ACCESS_KEY=your-secret-api-key
```

These map 1:1 with values visible in dashboard URLs and API key settings. [github](https://github.com/Tenderly/tenderly-docs/blob/master/simulations-and-forks/reference/configuration-of-api-access.md)

## Simulation examples

### 1. Single transaction simulation (no fork)

Use this to preview how a transaction would behave on a live network without broadcasting it. [docs.tenderly](https://docs.tenderly.co/reference)

```ts
import { Tenderly } from "@tenderly/sdk";

const tenderly = new Tenderly({
  accountName: process.env.TENDERLY_ACCOUNT!,
  projectSlug: process.env.TENDERLY_PROJECT!,
  accessKey: process.env.TENDERLY_ACCESS_KEY!,
  network: "mainnet",
});

export async function simulateTx(params: {
  from: string;
  to: string;
  data: string;
  value?: string;
}) {
  const simulation = await tenderly.simulations.simulateTransaction({
    networkId: "1", // mainnet chain id as string
    from: params.from,
    to: params.to,
    input: params.data,
    value: params.value ?? "0",
    gas: "auto",
    gasPrice: "auto",
  });

  // simulation.result includes status, logs, state diffs, gas usage, etc.
  return simulation;
}
```

Agent usage pattern:

- Before any real deploy or state-changing call, ask the agent to call `simulateTx`, inspect `status` and diffs, and **only proceed with deployment if the simulation succeeds**.

### 2. Fork creation + simulation

Use forks to chain multiple simulations and test complex flows on a private mainnet replica. [github](https://github.com/Tenderly/tenderly-docs)

```ts
import { Tenderly } from "@tenderly/sdk";

const tenderly = new Tenderly({
  accountName: process.env.TENDERLY_ACCOUNT!,
  projectSlug: process.env.TENDERLY_PROJECT!,
  accessKey: process.env.TENDERLY_ACCESS_KEY!,
  network: "mainnet",
});

export async function createForkAndSimulate() {
  // 1. Create a fork at a specific block
  const fork = await tenderly.forks.createFork({
    networkId: "1",
    blockNumber: "latest",
    alias: "hyperagent-dev-fork",
  });

  const forkId = fork.id;
  const forkRpcUrl = fork.rpcUrl;

  // 2. Use fork RPC in ethers/viem to simulate deployments and flows
  //    Example with ethers:
  /*
  const provider = new ethers.JsonRpcProvider(forkRpcUrl);
  const wallet = new ethers.Wallet(PRIVATE_KEY, provider);
  */

  // 3. You can also run simulations via API, scoped to this fork
  const sim = await tenderly.simulations.simulateTransaction({
    networkId: forkId, // fork identifier here
    from: "0x...",
    to: "0x...",
    input: "0x...",
  });

  return { fork, sim };
}
```

This pattern is ideal for HyperAgent:

- For each workflow run, create or reuse a fork.  
- Run a sequence of simulations (deploy → interact → verify) on the fork.  
- Only if everything passes, let the agent propose real mainnet transactions.

## REST API highlights

If you call the API directly (from Python workers, for example), key capabilities include: [docs.tenderly](https://docs.tenderly.co/reference/api)

- **Simulation API** – simulate single/bundled txs, inspect results.  
- **Fork API** – create, list, and manage forks / Virtual TestNets.  
- **Project operations** – list simulations, manage resources within a project.  
- **Webhooks & Actions** – trigger workflows when on-chain events match rules.

**Typical API access config:**

```env
TENDERLY_ACCOUNT=...
TENDERLY_PROJECT=...
TENDERLY_ACCESS_KEY=...
TENDERLY_BASE_URL=https://api.tenderly.co/api/v1
```

Example low-level call (Node):

```ts
import axios from "axios";

const tAxios = axios.create({
  baseURL: process.env.TENDERLY_BASE_URL!,
  headers: {
    "X-Access-Key": process.env.TENDERLY_ACCESS_KEY!,
  },
});

async function listProjectSimulations() {
  const account = process.env.TENDERLY_ACCOUNT!;
  const project = process.env.TENDERLY_PROJECT!;
  const res = await tAxios.get(`account/${account}/project/${project}/simulations`);
  return res.data;
}
```

## Integration guidance for AI / agents

When integrating Tenderly into an AI agent system (e.g., HyperAgent/ROMA):

- Treat Tenderly as the **canonical pre-flight check**:
  - Tools: `simulate_transaction`, `create_fork`, `simulate_on_fork`.  
  - Policy: never send a high-value on-chain transaction without at least one successful simulation.

- Prefer **fork-based workflows** for multi-step flows:
  - Agent should chain several simulations on a fork, inspect diffs and gas, then propose final actions.

- Be aware of:
  - Rate limits and plan-specific quotas (simulations, forks, alerts).  
  - API key security: never echo `TENDERLY_ACCESS_KEY` in logs or responses.

Let’s design a **TenderlyToolkit** assuming:

- HyperAgent / Studio side is **TypeScript/Node**.
- ROMA or other orchestrators can call it over HTTP (or you can port the same ideas to Python later).

Goal: a **small, deterministic API** your agents can call, not a thin wrapper over every Tenderly feature.

***

## 1. High-level shape

Expose just 3–4 tools:

- `createFork` – get an isolated Virtual TestNet to work on.  
- `simulateTransaction` – single-tx simulation (mainnet or fork).  
- `simulateBundle` (optional) – multi-step flow on a fork.  
- `inspectSimulation` – fetch details of an earlier simulation.

TypeScript interface:

```ts
// tenderly-toolkit/types.ts
export type NetworkId = string; // "1", "137", "43114", or fork ID

export interface CreateForkInput {
  networkId: NetworkId;
  alias?: string;
  blockNumber?: string; // "latest" or numeric as string
}

export interface CreateForkOutput {
  forkId: string;
  rpcUrl: string;
}

export interface SimulateTxInput {
  networkId: NetworkId;
  from: string;
  to?: string;
  input?: string;
  value?: string;
  gas?: "auto" | string;
  gasPrice?: "auto" | string;
}

export interface SimulateTxOutput {
  id: string;
  status: "success" | "reverted";
  gasUsed: string;
  blockNumber: string;
  transactionHash?: string;
  errorMessage?: string;
}

export interface TenderlyToolkit {
  createFork(input: CreateForkInput): Promise<CreateForkOutput>;
  simulateTransaction(input: SimulateTxInput): Promise<SimulateTxOutput>;
}
```

## 2. Implementation with Tenderly SDK (Node)

```ts
// tenderly-toolkit/index.ts
import { Tenderly } from "@tenderly/sdk";
import {
  CreateForkInput,
  CreateForkOutput,
  SimulateTxInput,
  SimulateTxOutput,
  TenderlyToolkit,
} from "./types";

function createClient() {
  return new Tenderly({
    accountName: process.env.TENDERLY_ACCOUNT!,
    projectSlug: process.env.TENDERLY_PROJECT!,
    accessKey: process.env.TENDERLY_ACCESS_KEY!,
    network: "mainnet",
  });
}

export function createTenderlyToolkit(): TenderlyToolkit {
  const client = createClient();

  return {
    async createFork(input: CreateForkInput): Promise<CreateForkOutput> {
      const fork = await client.forks.createFork({
        networkId: input.networkId,
        blockNumber: input.blockNumber ?? "latest",
        alias: input.alias ?? `hyperagent-fork-${Date.now()}`,
      });

      return {
        forkId: fork.id,
        rpcUrl: fork.rpcUrl,
      };
    },

    async simulateTransaction(input: SimulateTxInput): Promise<SimulateTxOutput> {
      const sim = await client.simulations.simulateTransaction({
        networkId: input.networkId,
        from: input.from,
        to: input.to,
        input: input.input ?? "0x",
        value: input.value ?? "0",
        gas: input.gas ?? "auto",
        gasPrice: input.gasPrice ?? "auto",
      });

      return {
        id: sim.id,
        status: sim.status === "succeeded" ? "success" : "reverted",
        gasUsed: sim.gasUsed,
        blockNumber: sim.blockNumber,
        transactionHash: sim.transactionHash,
        errorMessage: sim.errorMessage,
      };
    },
  };
}
```

Env:

```env
TENDERLY_ACCOUNT=your-account-or-org
TENDERLY_PROJECT=your-project-slug
TENDERLY_ACCESS_KEY=your-api-key
```

## 3. Expose as HTTP endpoints for ROMA / HyperAgent

If HyperAgent or ROMA wants to use this as a “tool server”, wrap it in a tiny Express / Fastify service:

```ts
// tenderly-service/server.ts
import express from "express";
import { createTenderlyToolkit } from "./tenderly-toolkit";

const app = express();
const toolkit = createTenderlyToolkit();

app.use(express.json());

app.post("/tenderly/create-fork", async (req, res) => {
  try {
    const result = await toolkit.createFork(req.body);
    res.json(result);
  } catch (err: any) {
    res.status(500).json({ error: err.message ?? "createFork failed" });
  }
});

app.post("/tenderly/simulate-tx", async (req, res) => {
  try {
    const result = await toolkit.simulateTransaction(req.body);
    res.json(result);
  } catch (err: any) {
    res.status(500).json({ error: err.message ?? "simulateTransaction failed" });
  }
});

app.listen(7001, () => {
  console.log("TenderlyToolkit service listening on :7001");
});
```

Then a ROMA/HyperAgent tool definition from Python or TS just calls these endpoints with clean JSON.

## 4. How an agent would use it (conceptually)

Inside your agent orchestration:

1. For a new “deploy & validate” run:
   - Call `/tenderly/create-fork` for the target network.  
   - Use `rpcUrl` with ethers/viem to deploy contracts on the fork.  
   - Call `/tenderly/simulate-tx` for any additional interactions the plan proposes.

2. Only if all simulations on the fork succeed:
   - Produce a final “on-chain execution plan” for the real network.  
   - Optionally simulate on live network without fork as a last safety check.

This keeps agents fast (your toolkit is a thin wrapper), deterministic (strict types and simple methods), and safe (everything goes through Tenderly before mainnet).
